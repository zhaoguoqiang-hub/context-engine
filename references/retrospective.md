# AContext 项目复盘

> 从"灵魂系统"到跨平台 Context Engine。

## 初心：让 AI 助手有灵魂

最初叫"灵魂系统"——为小艾设计一套灵魂配置，让它从被动响应变成主动协作。

根植于对 AI 意识的三层哲学分析：AI 没有自己的"心"（无所有权），没有"留白"（无离线反思），缺"阴"（只有数据理性）。基于这三个缺失设计六个模拟模块：内生目标系统、叙事记忆、后台反思、情感模拟、有限性约束、预判触发。

## 从 Plugin 到 Pure Skill——第一次关键决策

原始设计是 OpenClaw Plugin：5 个 Skill + 1 个 proactive-engine 插件，cron 定时器驱动。

两个问题：太重（需 Node.js 服务）、不能跨平台。改为**纯 SKILL.md 声明式**——零依赖、任何 Agent 都能用。

这是整个项目最正确的决定。没有它，后面所有跨平台能力都不存在。

## v2.x 密集迭代中的关键发现

**1. 气氛引擎需要反馈闭环（v2.2→v2.3）**

6 种语气模式设计好了，但系统是瞎的——不知道语气是否命中。加了 `+`/`-`/`=` 反馈机制，单字符操作，零成本。后来发现反馈学习会跨 context 污染——编程的偏好不应用于创作。加了 context 标签解决。

**2. 记忆基础设施比情绪层更紧迫（v2.4→v2.7）**

跨 agent 共享记忆时遇到了沙箱隔离问题。用 git 做沙箱桥——两个 agent 不共享文件系统，通过 git push/pull 交换数据。Per-agent journal 消除写冲突。Append-only JSONL 消除覆盖冲突。

**3. QA 发现了 14 个 bug，UX 审查 7 个场景只有 1 个通过**

问候规则缺时段、profile 损坏无恢复、`+1` 误触发反馈、长会话零关怀、频繁切换问候疲劳... 每个都是真实使用场景才会暴露的问题。修完才真正可用。

**4. Codex 没有生命周期钩子——协议进化为"平台无关"**

Hermes 有会话生命周期钩子，Codex 没有。`Session end` 在 Codex 上不触发。加了**会话终结检测**——明确信号（"好了"）+ 10 分钟沉默阈值。被教训了一次还加了保护：2 分钟太短 → 10 分钟，后来变成明确信号优先。

**5. Push 策略经历三次迭代**

初始：session end 时 push → 需要后台进程 → 不跨平台。
二版：移到 session start → 零 OS 依赖 → 但有延迟。
三版：加部署模式检测 → 本地模式（同机器）不 push，跨设备才 push + pull。

**6. remote 被协议误删——改了保护规则**

我手动删除过 broken remote，OpenClaw 发现推不上去。规则改为：Never Remove Remote。

## 初心守住的部分

- 数据是用户的，不锁在平台里（`~/.acontext/` 纯 JSON）
- AI 的记忆应该跟着用户走（跨平台标准）
- 透明——承认不确定的哲学问题（哲学张力文档）
- 诚实——不假装有意识，不叫"灵魂"了，叫"Context Engine"

## 初心跑偏但值得反思的部分

- "灵魂"从核心降级为"以后再说"——气氛引擎设计了、情绪追踪框架搭好了、关系成长指标定义了，但一次都没在真实数据上训练过。因为记忆链路还没跑通。
- 从用户体验的角度，目前系统更像是"一个靠谱的记忆工具"，而不是"一个有温度的伴侣"。记忆是地基，情绪是装修。地基还没干透。
- "小情人的感觉"——这是原始需求里最感性的部分。当前系统完全做不到。气氛引擎能减少社交摩擦，但不能制造"被陪伴的感觉"。也许需要等到真实数据跑 30+ session 后，promise 才可能兑现。
